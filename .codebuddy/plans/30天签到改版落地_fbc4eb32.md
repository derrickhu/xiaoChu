## User Requirements

- 现有签到页改为 **30 天一轮** 的完整签到方案，保留顶部累计进度条，并按签到进度推进。
- 下方奖励区按阶段切页：`1-7`、`8-14`、`15-21`、`22-28`，到 `29-30` 切换为终章奖励页。
- 第 7 天固定赠送 **SSR 宠物“炎狱火麟”**；若玩家已拥有，则改发对应碎片补偿。
- 灵石奖励需 **每日严格递增**，整体数值要比旧版更有爽感，并形成围绕炎狱火麟的培养闭环。
- 签到页底部新增 **“翻倍领奖”视频按钮**，带翻倍与视频标识；宠物本体不翻倍，只翻倍当天资源奖励。

## Product Overview

- 签到弹窗延续当前花花卷轴风格，顶部展示“累计签到 X 天”和 30 天进度条，中部按阶段显示每日奖励卡，底部显示主操作按钮与翻倍按钮。
- 页面会随签到天数切换不同 7 天奖励页，进度条在 `8/15/22/30` 节点形成明显阶段感，终章页强化 30 天收官氛围。
- 第 7 天奖励视觉需突出“炎狱火麟”与 SSR 价值，底部翻倍按钮固定在最下方操作区，点击后触发视频翻倍。

## Core Features

- 30 天签到轮次、累计进度、分页奖励区与终章页联动显示。
- 第 7 天真实发放炎狱火麟，并支持已拥有时的碎片补偿。
- 首轮高爽、后续折算的灵石递增奖励体系。
- 底部三态翻倍按钮：未签到弱提示、已签到可翻倍、已翻倍灰态。
- 奖励弹窗与签到提示文案同步支持“指定宠物”和“指定宠物碎片”展示。

## Tech Stack Selection

- 项目基于微信小游戏结构：`game.js / game.json`
- 现有实现为 **JavaScript + CommonJS**
- 签到页渲染为 **Canvas 2D 自绘弹窗**
- 继续复用现有能力：
- `js/views/dailyRewardView.js` 的签到绘制与点击分发
- `js/data/storage.js` 的签到状态与发奖
- `js/data/balance/economy.js` / `js/data/giftConfig.js` 的奖励配置

- `addToPetPool`、`addFragmentSmart`、`linesFromRewards` 等既有能力

## Implementation Approach

### 总体策略

在不推翻现有签到入口与弹窗框架的前提下，把当前“7 天奖励卡 + 30 天进度条”的混合实现，重构为“30 天轮次数据 + 按页渲染 + 真实发奖 + 底部翻倍按钮”的完整闭环。
数据层优先完成奖励模型、签到状态迁移和真实发奖；视图层只渲染当前页，避免一次绘制全部 30 天卡片，控制改动范围与渲染成本。

### 关键技术决策

1. **奖励模型继续沿用对象式 rewards，不新造活动系统**

- 当前 `LOGIN_REWARDS` 已是数组配置，适合直接扩为 30 条。
- 在原有 `soulStone / stamina / awakenStone / fragment` 基础上，补充“指定宠物”“指定宠物碎片”字段，避免再走旧 `petChoice` 假奖励语义。

2. **签到状态在 `storage.js` 做定向迁移，不做全局重置**

- 现有 `loginSign.day` 为 7 天循环字段，需平滑迁移到 30 天轮次。
- 采用 `_ensureLoginSign()` 定向补字段与兼容旧存档，保留 `totalSignDays` 与“今日是否已签到”，避免影响其他系统。
- 旧存档进入新系统时，保留已签进度并切入新 30 天首轮逻辑，避免玩家丢签到。

3. **广告翻倍复用现有 `signDouble`，但按钮升级为底部主操作区**

- 不新增广告位，不改 `AdManager` 主机制。
- 在签到页底部统一渲染按钮三态，并把“可翻倍资源”计算收口到存储/配置层，避免 UI 反推奖励时出现页码偏差或宠物误翻倍。

4. **炎狱火麟升级为 SSR 必须同步数值源与稀有度分组**

- `js/data/pets.js` 负责描述与品质分组。
- `js/data/balance/petBase.js` 才是真正的战斗面板来源。
- 两处同步调整，才能保证图鉴、培养成本、宠物池与战斗表现一致。

### Performance & Reliability

- 签到页渲染保持 **O(当前页可见卡片数)**，常规页 7 张卡、终章页 3 张卡，不改为 30 张卡全量渲染。
- 复用当前 `CHECKIN_HUAHUA` 资源与按钮底图；“视频”“x2”角标优先用 Canvas 文本/图形绘制，避免新增资源和 preload 依赖。
- 广告翻倍仅对当天资源生效，宠物本体与 30 天达成额外奖励不进入翻倍集合，防止重复发放 SSR。
- `claimLoginReward()` 应返回统一的“实际发放奖励 + 可翻倍奖励 + 页面状态”，避免视图层再按 `sign.day` 二次推导，减少 off-by-one 风险。

## Implementation Notes

- 必须清理 `dailyRewardView.js` 与 `adRewardPopup.js` 中旧 `petChoice / SSR自选 / SR自选` 文案，替换为固定宠物与碎片补偿语义。
- 现有正文文档与方案口径不一致处需同步修正：后续循环总灵石统一为 `3199`，并把底部翻倍按钮章节补成正式落地版。
- 优先把签到翻倍的“可翻倍资源计算”放到数据层，UI 只消费结果，避免广告补发逻辑与普通签到逻辑分叉。
- 避免牵连每日任务翻倍逻辑；本次只改签到页链路，降低 blast radius。
- 若不新增图片素材，则无需扩展 `js/main.js` 预加载；若后续改为独立视频小图标资源，再补 preload。

## Architecture Design

### 模块职责

- **奖励配置层**
- `js/data/balance/economy.js`
- `js/data/giftConfig.js`
- 负责 30 天奖励表、首轮/循环折算、可翻倍资源定义与页面分段辅助计算。

- **签到状态与发奖层**
- `js/data/storage.js`
- 负责旧存档迁移、30 天轮次推进、真实发放炎狱火麟、重复领取补碎片、广告翻倍补发。

- **签到视图层**
- `js/views/dailyRewardView.js`
- 负责顶部进度条、当前阶段卡片页、终章页、底部主按钮与翻倍按钮三态渲染、点击行为分发。

- **奖励展示层**
- `js/views/adRewardPopup.js`
- 负责把固定宠物、固定宠物碎片和资源翻倍行准确展示给玩家。

- **宠物定义层**
- `js/data/pets.js`
- `js/data/balance/petBase.js`
- 负责把炎狱火麟从 SR 升为 SSR，并同步基础面板与品质归属。

### 数据关系

- `economy.js` 定义 30 天奖励
- `giftConfig.js` 对外导出奖励与分页辅助
- `storage.js` 消费奖励配置并发奖
- `dailyRewardView.js` 消费 `storage + giftConfig` 渲染当前页
- `adRewardPopup.js` 消费实际奖励结果渲染翻倍弹窗
- `pets.js / petBase.js` 提供炎狱火麟的品质、显示与战斗数值

## Key Code Structures

- `LOGIN_REWARDS` 奖励项建议统一为“可扩展奖励对象”，至少支持：
- `soulStone`
- `stamina`
- `awakenStone`
- `fragment`
- `petId`
- `petFragment: { petId, count }`
- `claimLoginReward()` 返回值建议至少包含：
- 当前领取天数/当前轮页信息
- 实际发放奖励
- 可翻倍奖励
- 是否发放整宠 / 是否走重复补偿
- 视图层底部按钮状态建议统一成：
- `canSign`
- `canDouble`
- `doubledDone`
- `hintOnly`

## 设计方向

采用国风卷轴签到弹窗，延续现有花花资源的轻仙侠气质，但把结构升级成真正的 30 天页面。顶部是卷轴标题与累计文案，中部是 30 天进度条与阶段节点，下方是按页切换的奖励卡组，底部是固定操作区。

## 页面区块

- **标题区**：卷轴横幅居中，“每日奖励”保留金边描边字，副文案显示累计天数与当前轮天数。
- **进度区**：奶白底槽 + 绿色填充，`8/15/22/30` 礼盒节点悬浮在轨道上方，当前阶段节点发亮。
- **奖励区**：`1-28` 天保持 6 小卡 + 1 宽卡的熟悉结构；`29-30` 切终章页，用双大卡 + 达成宽卡增强收官仪式感。
- **操作区**：底部主按钮始终占视觉中心；翻倍按钮固定在最下方，显示“翻倍领奖”与视频/x2 标识，已翻倍后转灰。

## Agent Extensions

### SubAgent

- **code-explorer**
- Purpose: 复核签到链路涉及的配置、存储、视图与宠物模块调用关系，确认改动边界和回归点。
- Expected outcome: 输出可靠的影响面结论，确保 30 天签到、真实发宠和翻倍按钮改造不漏掉关键调用点。
