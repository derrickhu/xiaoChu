## 一、项目概述

- **项目名称**：灵宠消消塔（五行通天塔）
- **项目类型**：微信小游戏（Canvas 2D）
- **核心玩法**：Roguelike 爬塔 + 智龙迷城式拖拽转珠 + 五行克制
- **核心特点**：无局外养成，死亡即重开，仅记录最高层数
- **技术栈**：原生 JavaScript + Canvas 2D + 微信云开发
- **构建工具**：无 Webpack/Vite，直接使用微信开发者工具编译
- **AppID**：`wx53b03390106eff65`

## 三、架构设计

### 3.1 整体架构模式

项目采用 **单例 Main + 场景驱动 + 委托分发** 模式：

```
                          ┌─────────────────────────┐
                          │     Main（js/main.js）    │
                          │  ┌─────────────────────┐ │
                          │  │ 全部游戏状态（g.*）   │ │
                          │  └─────────────────────┘ │
                          │  loop: update() → render()│
                          └────┬──────┬──────┬───────┘
                               │      │      │
              ┌────────────────┤      │      ├────────────────┐
              ▼                ▼      ▼      ▼                ▼
       ┌────────────┐  ┌──────────┐ ┌──────┐ ┌──────────┐ ┌────────┐
       │  engine/*   │  │  views/* │ │data/*│ │  input/* │ │runtime/│
       │(战斗/技能/  │  │(场景渲染/│ │(配置/│ │(触摸处理)│ │(音频)  │
       │ 动画/Run/   │  │ 弹窗/UI) │ │存储) │ │         │ │        │
       │ 教学)       │  │         │ │      │ │         │ │        │
       └────────────┘  └──────────┘ └──────┘ └──────────┘ └────────┘
```

**核心设计要点**：

1. **单例 Main**：所有游戏状态集中在 `Main` 类的实例 `g` 中，各模块通过接收 `g` 参数读写共享状态
2. **场景驱动**：`g.scene` 控制当前场景，`render()` 和 `onTouch()` 通过 `switch(this.scene)` 分发到对应模块
3. **委托模式**：Main 提供大量桩方法（`_rXxx`/`_tXxx`/`_applyXxx` 等），内部转发到具体模块
4. **无事件总线**：模块间通过直接方法调用和共享 `g` 对象通信

## 四、核心模块设计

### 4.1 渲染模块（`js/render.js`）

**核心类**：`Render`

**职责**：Canvas 2D 绘制封装、图片缓存管理、UI 组件绘制

**关键设计**：
- **图片缓存**：`_imgCache` 对象缓存已加载的 `wx.createImage()` 实例，`getImg(path)` 懒加载并缓存
- **预加载**：`preloadImages(paths, onProgress)` 返回 Promise，5 秒超时保底
- **主题色系统**：全局 `TH` 对象定义颜色方案（bg/card/text/accent/danger/success 等），所有模块统一引用
- **适配方案**：`S = canvasWidth / 375` 缩放因子，所有尺寸乘以 `S` 实现等比适配；`safeTop` 处理安全区域
- **ViewEnv 共享**：`js/views/env.js` 提供 `ctx/R/TH/W/H/S/safeTop/COLS/ROWS` 的 getter，供所有 view 模块访问

**属性配色系统**（`tower.js` 中定义，渲染模块引用）：

| 属性 | 主色 | 用途 |

| 金(metal) | `#ffd700` | 珠子、伤害文字、属性标识 |

### 4.2 战斗引擎（`js/engine/battle.js`）

#### 4.2.1 战斗状态机

```
playerTurn ──(松手/超时)──→ elimAnim ──(16帧/组)──→ dropping ──(10帧)──→
     ↑                                                      │
     │    ←── 有新的连消 ←──────────────────────────────────┘
     │                                                      │
     │    ←── preEnemy(30帧) ← enemyTurn ← 无连消且 combo=0 ┘
     │                                                      │
     │    ←── petAtkShow(38帧) ← 无连消但 combo>0 ──────────┘
     │              │
     │              ▼
     │         preAttack(12帧) → executeAttack → settle
     │              │
     │              ▼
     └──── preEnemy(30帧) → enemyTurn ──→ playerTurn
```

**状态说明**：
- `playerTurn`：玩家拖拽转珠阶段，8 秒限时（`dragTimeLimit = 8*60` 帧 @60fps）
- `elimAnim`：逐组消除动画，每组 16 帧
- `dropping`：珠子下落填充，10 帧后再检测连消
- `petAtkShow`：宠物攻击数值翻滚展示，38 帧
- `preAttack`：攻击结算前等待，12 帧
- `preEnemy`：敌方回合前等待，30 帧
- `enemyTurn`：敌人攻击和技能释放
- `victory`/`defeat`/`adReviveOffer`：胜负判定

#### 4.2.2 消除算法（`findMatchesSeparate`）

**三步流程**：
1. **横向扫描**：逐行查找连续 3+ 颗同色珠子并标记
2. **纵向扫描**：逐列查找连续 3+ 颗同色珠子并标记
3. **BFS 连通分组**：将标记的格子按属性做 BFS，合并相邻同属性为连通块

**返回值**：`[{ attr, count, cells: [{r,c}] }]`

**防初始三连**：`wouldMatch()` 在 `initBoard()` 和 `fillBoard()` 中确保新珠子不会立即形成三连

#### 4.2.3 伤害计算公式

```
最终伤害 = 基础攻击 × 消除倍率 × Combo倍率 × 全局加成 × 属性加成
           × 五行克制 × 暴击 × 残血爆发 × 法宝加成 - 敌人防御
```

**各因子详解**：

| 因子 | 计算方式 |
|------|----------|
| 基础攻击 | `getPetStarAtk(pet)` × 消除倍率 × `(1 + runBuffs.allAtkPct/100)` |
| 消除倍率 | 3消=1.0，4消=1.5，5消+=2.0（受 `elim3/4/5DmgPct` 加成） |
| Combo倍率 | 1-8段: `1 + (combo-1)*0.35`，8-12段: `+0.20/段`，12+段: `+0.10/段` |
| 五行克制 | 克制 `×2.5`，被克 `×0.5`（受 `counterDmgPct` 加成） |
| 暴击 | 暴击率由技能/法宝/buff 提供；暴击伤害=基础 `50%` + 加成 |
| 残血爆发 | HP≤15%: `×2.0`，HP≤30%: `×1.5` |
| 防御 | `dmg = max(0, dmg - enemy.def)` |

#### 4.2.4 棋盘管理

- **尺寸**：5 行 × 6 列（代码中 `ROWS=5, COLS=6`）
- **数据结构**：`g.board[r][c] = { attr: string, sealed: boolean|number }`
- **灵珠权重**：通过 `getBeadWeights()` 动态调整各属性珠子出现概率，敌人属性珠权重 ×1.4，法宝 `beadRateUp` 指定属性 ×1.5

### 4.3 Run 管理（`js/engine/runManager.js`）

**核心职责**：Roguelike 局内生命周期管理

```
startRun(g)         新局初始化（宠物池、初始队伍、法宝、runBuffs）
  │
  ▼
nextFloor(g)        层数+1、境界成长、生成事件
  │
  ▼
curEvent 分发 ──→ 战斗/奇遇/商店/休息
  │
  ▼
奖励/强化 ──────→ nextFloor (循环)
  │
  ▼ （floor > MAX_FLOOR 或 defeat）
endRun(g)           记录最高层、提交排行、清除存档
```

**关键接口**：

| 函数 | 说明 |
|------|------|
| `startRun(g)` | 初始化局内状态，生成宠物池、初始队伍(5只)、法宝(1件)，首次游戏触发教学 |
| `nextFloor(g)` | 层数+1，境界成长(加血量上限)，每5层隐性加攻击，生成事件 |
| `endRun(g)` | 更新最高层记录，清除存档，提交排行榜，跳转结算 |
| `saveAndExit(g)` | 深拷贝局内全部状态到 `storage.savedRun`，回到标题 |
| `resumeRun(g)` | 从存档恢复状态，清除存档，跳转事件场景 |
| `onDefeat(g)` | 失败处理链：临时复活→额外复活→法宝不灭金身→分享复活→真正失败 |
| `doAdRevive(g)` | 调用 `wx.shareAppMessage` 分享后满血复活 |

**runBuffs 结构**（`DEFAULT_RUN_BUFFS`）：

```javascript
{
  allAtkPct, allDmgPct, attrDmgPct:{metal,wood,earth,water,fire},
  heartBoostPct, weaponBoostPct, extraTimeSec,
  hpMaxPct, comboDmgPct, elim3DmgPct, elim4DmgPct, elim5DmgPct,
  counterDmgPct, skillDmgPct, skillCdReducePct, regenPerTurn,
  dmgReducePct, bonusCombo, stunDurBonus,
  enemyAtkReducePct, enemyHpReducePct, enemyDefReducePct,
  eliteAtkReducePct, eliteHpReducePct, bossAtkReducePct, bossHpReducePct,
  nextDmgReducePct, postBattleHealPct, extraRevive
}
```

**道具系统**（`runManager.js` 中实现）：
- 乾坤重置：分享获取，使用后重置棋盘 (`initBoard`)
- 回春妙术：分享获取，使用后满血恢复
- 每局各限获取和使用一次

### 4.5 动画系统（`js/engine/animations.js`）

**驱动方式**：帧驱动（每帧 `update()` 调用各动画更新函数）

**动画类型**：

| 动画 | 数据结构 | 更新方式 |
|------|----------|----------|
| 屏幕震动 | `g.shakeT/shakeI` | 每帧递减 |
| Combo 弹出 | `g._comboAnim` | 缩放弹跳 + 透明度渐隐 |
| 粒子特效 | `g._comboParticles[]` | 物理运动(速度+重力) + 生命周期过滤 |
| 伤害飘字 | `g.dmgFloats[]` | 分三段上飘：停留→正常→加速消失 |
| 技能特效 | `g.skillEffects[]` | 上飘 + 缩放弹跳 + 透明度渐隐 |
| 消除飘字 | `g.elimFloats[]` | 弹入→停留→上飘→消失 |
| 数值翻滚 | `g.petAtkNums[]` | 从0快速翻滚到最终值 |
| 交换动画 | `g.swapAnim` | 线性插值位移 |
| 战斗动画 | `heroAttackAnim/enemyHurtAnim` 等 | 进度 0→1 控制 |
| HP 变化 | `_enemyHpLoss/_heroHpLoss/_heroHpGain` | 从旧比例动画到新比例 |
| 敌人死亡 | `_enemyDeathAnim` | 爆裂粒子特效 |
| Boss 入场 | `_bossEntrance` | 帧数递减 |

### 4.6 输入系统（`js/input/touchHandlers.js`）

**核心模式**：按场景分发，每个场景有独立的 `tXxx(g, type, x, y)` 处理函数

**转珠操作**：
1. `touchstart`：判断触点落在棋盘格内，记录 `dragR/dragC/dragAttr`
2. `touchmove`：计算目标格，若为相邻格（八方向）且非封印，立即交换 `board[or][oc] ↔ board[r][c]`
3. `touchend`：调用 `g._checkAndElim()` 触发消除链

**宠物技能释放**：
- 上滑宠物头像：`_petSwipeIndex` 记录起始位置，滑动距离超过阈值触发 `g._triggerPetSkill`
- 长按宠物头像：`_petLongPressTimer` 计时 300ms 后显示技能预览浮窗

### 4.7 音频系统（`js/runtime/music.js`）

**核心类**：`MusicManager`（导出为 `MusicMgr` 单例）

## 六、存储与云服务设计

### 6.1 双重存储策略

```
                ┌──────────────┐
                │  Storage 类  │
                └──┬────────┬──┘
                   │        │
         ┌─────────▼──┐  ┌──▼──────────┐
         │ 本地存储    │  │ 云数据库     │
         │ wx.storage  │  │ playerData  │
         │ (即时写入)  │  │ (2秒防抖)   │
         └────────────┘  └─────────────┘
```

- **本地存储 key**：`wxtower_v1`
- **云环境 ID**：`cloud1-6g8y0x2i39e768eb`
- **写入策略**：每次 `_save()` 先写本地，再 2 秒防抖同步到云端
- **冲突处理**：`_syncFromCloud()` 比较 `_updateTime` 时间戳，云端较新时深度合并到本地；数值字段智能合并（bestFloor 取大，bestTotalTurns 取小）

### 6.2 云函数

| 云函数 | 说明 | 调用时机 |
|--------|------|----------|
| `getOpenid` | 获取用户 openid | Storage 初始化时 |
| `initCollections` | 创建 `playerData`/`rankAll`/`rankDex`/`rankCombo` 集合 | Storage 初始化时 |
| `ranking` | 排行榜提交与查询，支持 6 种 action | 局结束提交 + 排行榜页拉取 |
| `resetTaskAndWeekly` | 每日/每周任务重置 | **前端未接入** |

### 6.3 排行榜设计

**三个独立榜单**：

| 榜单 | 云集合 | 排序逻辑 | 提交时机 |
|------|--------|----------|----------|
| 速通榜 | `rankAll` | floor DESC → totalTurns ASC | 局结束时 |
| 图鉴榜 | `rankDex` | petDexCount DESC | 局结束时/独立提交 |
| 连击榜 | `rankCombo` | maxCombo DESC | 局结束时/独立提交 |

**缓存策略**：
- 30 秒客户端缓存（`rankLastFetch`），避免频繁调用
- 后台预热：Storage 初始化完成后静默拉取速通榜
- 排行榜页面每 2 分钟自动刷新（`af % 7200 === 0`）

### 6.4 用户授权

- 使用 `wx.createUserInfoButton` 创建透明按钮覆盖在 Canvas 排行按钮上
- 授权后保存 `{ nickName, avatarUrl }` 到本地
- 过滤无效默认值（`微信用户`）
- 授权失败引导至设置页重新开启

## 九、关键技术要点备忘

### 9.1 微信小游戏适配

- **Canvas 创建**：`GameGlobal.__mainCanvas` 复用 `game.js` 中创建的首个 Canvas（屏幕 Canvas）
- **DPR 适配**：`canvas.width = windowWidth * pixelRatio`，所有绘制坐标乘以 `dpr`
- **安全区域**：`safeTop = (safeArea?.top || 20) * dpr`，UI 内容从安全区域下方开始绘制
- **缩放因子**：`S = canvasWidth / 375`，以 375pt 宽度为基准等比缩放

### 9.2 触摸坐标转换

```javascript
const x = touch.clientX * dpr
const y = touch.clientY * dpr
```

触摸事件的 `clientX/Y` 是逻辑像素，需乘以 `dpr` 转换为 Canvas 像素坐标。

### 9.3 原生按钮覆盖

微信小游戏的 `UserInfoButton` 和 `FeedbackButton` 是原生 UI 组件，需要：
- 在 Canvas 对应位置创建透明的原生按钮
- 随场景切换动态创建/销毁
- 按钮位置需从 Canvas 像素换算回 CSS 像素（`÷ dpr`）

### 9.4 宠物池机制

每局 `startRun` 时通过 `generateSessionPetPool()` 生成本局宠物池（每属性 5 只，共 25 只），后续所有宠物获取都从此池中抽取，保证局内不出现重复宠物。
